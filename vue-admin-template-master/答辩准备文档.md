## 一、系统概述

### 1. 项目定位
本系统是一个**寻路算法可视化教学平台**，旨在通过可视化的方式帮助学生直观理解 BFS、DFS、Dijkstra、A* 四种经典寻路算法的执行过程。

### 2. 技术栈
| 层级 | 技术 | 用途 |
|------|------|------|
| **前端** | Vue 2 + Element UI | 页面构建与UI组件 |
| **前端路由** | Vue Router | 页面路由管理 |
| **状态管理** | Vuex | 用户登录状态管理 |
| **HTTP请求** | Axios | 前后端数据交互 |
| **图表** | ECharts | 管理仪表板数据可视化 |
| **后端** | Spring Boot + MyBatis-Plus | RESTful API 服务 |
| **数据库** | MySQL (pathfinding) | 数据持久化 |
| **缓存** | Redis | 数据缓存 |
| **认证** | JWT | 用户身份认证 |
| **API文档** | SpringDoc OpenAPI (Swagger) | 接口文档 |

## 三、答辩常见问题及回答

### 问题1：请介绍一下你的毕设项目做了什么？

**回答：**
我的毕业设计题目是"寻路算法可视化教学平台"。它是一个基于 Vue.js 和 Spring Boot 的 Web 应用，主要功能是帮助计算机专业的学生直观地学习四种经典寻路算法：广度优先搜索（BFS）、深度优先搜索（DFS）、Dijkstra 算法和 A* 算法。

系统分为学生端和管理员端。学生可以在可视化页面上设置起点、终点和障碍物，选择算法后系统会以动画形式展示算法的搜索过程和最终路径。同时还有算法学习页面提供原理讲解和伪代码，以及实验记录功能保存每次实验的数据。管理员端则提供了用户管理、题库管理和实验记录查看等功能。

### 问题2：为什么选择 Vue 而不是 React？Vue 有什么特点？

**回答：**
选择 Vue 主要有几个原因：
1. **上手门槛低**：Vue 的模板语法更接近传统 HTML，学习曲线平缓
2. **双向数据绑定**：通过 `v-model` 指令可以轻松实现表单数据的双向绑定，这在我们的实验记录编辑、筛选表单等场景中非常方便
3. **组件化开发**：在可视化页面中，我将网格地图（GridBoard）、控制面板（ControlPanel）、步骤信息（StepInfo）拆分为独立组件，便于维护和复用
4. **生态系统完善**：配合 Element UI 组件库可以快速搭建后台管理界面

**Vue 的核心特点**：
- **响应式数据**：Vue 使用基于依赖追踪的响应式系统，当数据变化时视图自动更新
- **指令系统**：如 `v-for` 渲染列表、`v-if` 条件渲染、`v-bind` 绑定属性
- **单文件组件**：将模板、脚本、样式写在同一个 `.vue` 文件中

### 问题3：Vue Router 在项目中是怎么用的？路由守卫怎么配置的？

**回答：**
Vue Router 用于管理页面之间的导航。在项目中，我使用了**路由元信息（meta）** 来控制页面权限：

```javascript
// 路由配置示例
{
  path: '/visualization',
  component: Layout,
  children: [{
    path: 'index',
    component: () => import('@/views/visualization/index'),
    name: 'Visualization',
    meta: {
      title: '算法可视化',
      icon: 'el-icon-video-play',
      roles: ['student', 'admin'] // 权限控制
    }
  }]
}
```

**路由守卫**在 `permission.js` 中配置：
- **前置守卫** `router.beforeEach`：检查用户是否登录，未登录则跳转到登录页
- **角色验证**：根据 `meta.roles` 判断用户是否有权限访问该页面
- **动态路由**：管理员和学生看到不同的侧边栏菜单

### 问题4：Axios 在项目中是怎么封装的？如何处理 token？

**回答：**
我在 `src/utils/request.js` 中对 Axios 进行了封装：

1. **请求拦截器**：在每次请求前，从 localStorage 获取 JWT token，添加到请求头的 `Authorization` 字段
2. **响应拦截器**：对后端返回的数据进行统一处理，提取 `res.data` 字段，让组件直接拿到业务数据
3. **错误处理**：统一处理 401（未授权跳转登录）、403（权限不足）、500（服务器错误）等状态码

```javascript
// 请求拦截器
service.interceptors.request.use(config => {
  const token = getToken()
  if (token) {
    config.headers['Authorization'] = 'Bearer ' + token
  }
  return config
})

// 响应拦截器
service.interceptors.response.use(
  response => {
    const res = response.data
    return res.data // 直接返回业务数据
  },
  error => {
    // 统一错误处理
  }
)
```

### 问题5：算法可视化是怎么实现的？前端如何执行算法？

**回答：**
算法是在前端 JavaScript 中直接执行的，没有依赖后端。具体实现：

1. **算法逻辑**：在 `src/views/visualization/algorithms.js` 中实现了 BFS、DFS、Dijkstra、A* 四种算法
2. **执行流程**：
   - 用户设置起点、终点、障碍物，选择算法
   - 点击"开始运行"后，调用 `runAlgorithm()` 函数
   - 算法返回 `visitedOrder`（访问顺序）和 `path`（最短路径）
3. **动画播放**：使用 `async/await` 和 `setTimeout` 实现逐帧动画，按顺序逐个高亮显示访问的节点和最终路径
4. **速度控制**：通过调整延迟时间（1ms~200ms）控制动画速度

### 问题6：四种寻路算法的区别是什么？各自的时间复杂度？

**回答：**

| 算法 | 核心思想 | 数据结构 | 时间复杂度 | 特点 |
|------|---------|---------|-----------|------|
| **BFS** | 逐层扩展，先访问距离起点近的节点 | 队列（FIFO） | O(V+E) | 无权图保证最短路径 |
| **DFS** | 沿一条路径走到底，回溯继续 | 栈/递归 | O(V+E) | 不保证最短路径 |
| **Dijkstra** | 贪心策略，每次选距离最小的节点 | 优先队列（最小堆） | O((V+E)logV) | 带权图保证最短路径 |
| **A*** | 启发式搜索，f(n)=g(n)+h(n) | 优先队列 | O(b^d) | 有启发函数引导，效率高 |

**在项目中的使用**：可视化页面中，用户选择不同算法后，系统会以不同颜色的动画展示搜索过程。BFS 像水波一样扩散，DFS 像一条线深入探索，Dijkstra 和 A* 则根据距离/估价函数优先探索更有希望的节点。

### 问题7：前后端是怎么交互的？接口是怎么设计的？

**回答：**
前后端通过 RESTful API 进行交互，使用 JSON 格式传输数据。

**接口设计规范**：
- 统一前缀 `/api`
- 统一响应格式 `R<T>`：包含 `code`、`message`、`data` 三个字段
- 分页接口返回 `Page<T>`：包含 `records`、`total`、`current`、`size`

**主要接口示例**：
```
POST   /api/user/login          # 用户登录
POST   /api/user/register       # 用户注册
GET    /api/user/{id}           # 获取用户信息
PUT    /api/user/{id}           # 更新用户信息
GET    /api/user/list           # 用户列表（分页）

POST   /api/experiment          # 创建实验记录
GET    /api/experiment/list     # 实验记录列表（分页）
GET    /api/experiment/{id}     # 实验记录详情
GET    /api/experiment/{id}/steps  # 实验步骤

GET    /api/problem/list        # 题目列表（分页）
POST   /api/problem             # 新增题目
PUT    /api/problem/{id}        # 更新题目
DELETE /api/problem/{id}        # 删除题目

POST   /api/problem-completion  # 保存题目学习记录
GET    /api/problem-completion/user/{userId}  # 用户学习记录
```

### 问题8：数据库是怎么设计的？有哪些核心表？

**回答：**
数据库使用 MySQL，核心表包括：

1. **user 表**：用户信息（userId, username, password, name, studentId, email, role, createdAt）
2. **algorithm 表**：算法信息（algorithmId, name, description, complexity）
3. **problem 表**：题目信息（problemId, title, description, difficulty, algorithmId, leetcodeLink）
4. **experiment_record 表**：实验记录（recordId, userId, algorithmId, startTime, endTime, visitedCount, pathLength, status）
5. **experiment_step 表**：实验步骤（stepId, recordId, stepIndex, nodeX, nodeY, nodeState）
6. **problem_completion 表**：题目学习记录（completionId, userId, problemId, completionStatus, isSolved, attemptCount, notes, difficultyRating, timeSpentMinutes, lastReviewDate）

### 问题9：JWT 认证是怎么实现的？

**回答：**
JWT（JSON Web Token）用于用户身份认证：

1. **登录时**：后端验证用户名密码后，生成 JWT token 返回给前端
2. **前端存储**：前端将 token 存储在 localStorage 中
3. **请求时**：Axios 请求拦截器在每次请求的 Header 中添加 `Authorization: Bearer <token>`
4. **后端验证**：Spring Boot 通过 JWT 过滤器解析 token，验证用户身份
5. **token 过期**：token 有效期 24 小时，过期后需要重新登录

### 问题10：实验记录的重放功能是怎么实现的？

**回答：**
实验重放功能的实现流程：

1. **保存**：算法执行时，每访问一个节点或确定路径上的一点，都会记录步骤（stepIndex, nodeX, nodeY, nodeState）
2. **存储**：实验完成后，将实验记录和所有步骤保存到数据库
3. **加载**：在实验记录页面点击"重放实验"，跳转到可视化页面并传入 `experimentId`
4. **回放**：可视化页面加载时，从 URL 参数获取 `experimentId`，调用 API 获取实验步骤，然后按顺序在网格上展示

### 问题13：项目中遇到了哪些难点？怎么解决的？

**回答：**
主要难点和解决方案：

1. **算法动画性能优化**：当网格较大（如78x64）时，逐帧更新所有节点会导致页面卡顿。解决方案是采用**批量更新**策略，每批处理多个节点后再更新视图，减少 Vue 响应式系统的更新次数。

2. **前后端数据格式不一致**：后端返回的数据结构在不同接口中不一致。解决方案是在 Axios 响应拦截器中统一处理，并在每个页面中增加多种数据格式的兼容处理。

3. **实验步骤的保存与回放**：需要保存算法执行过程中的每一步状态。解决方案是设计 `experiment_step` 表，记录每个步骤的坐标和状态（visited/path），回放时按顺序加载。

4. **权限控制**：学生和管理员看到不同的页面。解决方案是使用 Vue Router 的 `meta.roles` 字段和路由守卫进行权限验证。

### 问题14：Vuex 在项目中用来做什么？

**回答：**
Vuex 主要用于管理用户的登录状态：
- **user 模块**：存储用户信息（token、角色、用户名等）
- **登录/登出**：通过 `dispatch` 调用 Vuex 的 action 进行登录和登出操作
- **持久化**：页面刷新时，从 localStorage 恢复用户状态
