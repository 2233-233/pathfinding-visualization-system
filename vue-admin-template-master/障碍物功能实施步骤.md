### 第一阶段：数据库实施

#### 1.1 创建Obstacle表
```sql
-- 在MySQL中执行以下SQL语句
CREATE TABLE `Obstacle` (
  `obstacle_id` INT PRIMARY KEY AUTO_INCREMENT,
  `record_id` INT NOT NULL,
  `x` INT NOT NULL COMMENT '障碍物X坐标',
  `y` INT NOT NULL COMMENT '障碍物Y坐标',
  `obstacle_type` ENUM('wall', 'water', 'mountain') DEFAULT 'wall' COMMENT '障碍物类型（可扩展）',
  `created_at` DATETIME DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY (`record_id`) REFERENCES `ExperimentRecord`(`record_id`)
    ON DELETE CASCADE ON UPDATE CASCADE,
  INDEX `idx_record_id` (`record_id`),
  INDEX `idx_coordinates` (`x`, `y`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='实验障碍物坐标表';
```

#### 1.2 数据迁移（可选）
如果有现有的实验记录数据，需要从ExperimentStep表的gridState字段中提取障碍物信息：
```sql
-- 创建临时表存储从gridState提取的障碍物
CREATE TEMPORARY TABLE temp_obstacles AS
SELECT 
  es.record_id,
  JSON_EXTRACT(es.grid_state, '$.obstacles[*].x') as x,
  JSON_EXTRACT(es.grid_state, '$.obstacles[*].y') as y
FROM ExperimentStep es
WHERE es.grid_state IS NOT NULL
  AND JSON_EXTRACT(es.grid_state, '$.obstacles') IS NOT NULL;

-- 插入到Obstacle表
INSERT INTO Obstacle (record_id, x, y, obstacle_type)
SELECT record_id, x, y, 'wall'
FROM temp_obstacles;

-- 删除临时表
DROP TEMPORARY TABLE temp_obstacles;
```

## 关键注意事项

### 1. 向后兼容
- 确保现有功能不受影响
- 如果现有数据没有障碍物信息，接口应返回空数组或默认值

### 2. 性能考虑
- 一个实验可能有很多障碍物，批量操作时注意性能
- 考虑使用分页查询障碍物列表（如果数量很大）

### 3. 数据一致性
- 障碍物与实验记录的生命周期保持一致
- 删除实验记录时，相关的障碍物应自动删除（通过外键约束）

### 4. 错误处理
- 添加适当的错误处理和用户提示
- 处理网络异常和服务器错误

### 5. 扩展性考虑
- obstacle_type字段设计为ENUM，便于后续扩展障碍物类型
- 可以考虑添加障碍物权重、颜色等扩展属性

## 风险与应对

### 风险1：性能问题
- **风险描述**：大量障碍物可能导致性能下降
- **应对措施**：
  - 使用批量操作减少数据库交互
  - 添加适当的数据库索引
  - 考虑分页查询障碍物列表

### 风险2：数据迁移问题
- **风险描述**：现有数据迁移可能失败或丢失数据
- **应对措施**：
  - 先备份现有数据
  - 在测试环境充分测试迁移脚本
  - 提供回滚方案

### 风险3：前端兼容性问题
- **风险描述**：修改现有页面可能影响其他功能
- **应对措施**：
  - 充分测试现有功能
  - 使用版本控制，便于回退
  - 逐步部署，先测试环境后生产环境
